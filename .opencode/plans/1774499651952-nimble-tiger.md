### Tooling and environment
- Python requirement in `pyproject.toml`: `>=3.12`.

- Poetry is the documented setup/run entrypoint; `tool.poetry.requires-poetry = ">=2.1"`.

- Official Poetry docs confirm `poetry install` and `poetry run ...` are the right repo-level commands for this setup.

### Commands actually available in the repo
- Install/setup: `poetry install`
- Run scripts: `poetry run python <script>.py`
- Best repo-wide validation command: `pre-commit run --all-files`
- Individual formatter/linter commands implied by config:
  - `poetry run isort .`
  - `poetry run black .`
  - `poetry run flake8 --format pylint .`
  - `poetry run pylint --output-format parseable .`
  - `poetry run prettier --check .`
- No dedicated build command is configured; AGENTS.md should say that explicitly.

### Testing reality
- There is no formal unit-test runner configured in the repo.
- The main validation scripts are executable integration/manual scripts:
  - `poetry run python ./test_api.py`
  - `poetry run python ./test_c1000x_mqtt_controls.py`
- Focused script execution is possible for utilities such as:
  - `poetry run python ./monitor.py --site-id <SITE_ID>`
  - `poetry run python ./monitor.py --device-id <DEVICE_ID>`
  - `poetry run python ./mqtt_monitor.py --device-sn <DEVICE_SN>`
- `test_api.py` uses internal boolean toggles (`TESTAUTHENTICATE`, `TESTAPIMETHODS`, `TESTAPIENDPOINTS`, `TESTAPIFROMJSON`) rather than CLI single-test selectors.
- AGENTS.md must therefore explain that there is no pytest-style single-test command today; the closest equivalent is running one integration script or narrowing a utility script target.

### Style and design conventions to document
- Formatting is driven by Black and isort (`profile = "black"`).
- Flake8 ignores: `E501,E226,W503,E231`.
- Pylint disables: `import-error`, `line-too-long`, `invalid-name`, `protected-access`.
- Typical import grouping is: stdlib, blank line, third-party, blank line, local imports.
- `from __future__ import annotations` is common in core modules.
- `TYPE_CHECKING` is used for type-only imports in some modules.
- Internal helpers typically use `snake_case`, but public API parameters/methods often preserve compatibility-oriented camelCase names such as `countryId`, `siteId`, `deviceSn`, `fromFile`, `testDir`, `logLevel`, and `endpointLimit`.
- Error mapping should reuse `api/errors.py` rather than introducing parallel exception handling.
- Request/network logic should follow the retry/logging/session patterns already used in `api/session.py`.
- Payload normalization/cache update code often logs and continues when one field is malformed instead of aborting the whole update.
- Module/class/function docstrings are common and should be preserved.
- Logging conventions include `_LOGGER` for module loggers and `CONSOLE` for user-facing script output.
